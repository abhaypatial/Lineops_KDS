## Project overview

LineOps KDS is a production-ready, multi-tenant Kitchen Display System (KDS) built as a
pnpm monorepo. It handles real-time order management for commercial kitchens via WebSocket
push updates, a React display frontend, and an Express REST/WS backend.

---

## Monorepo structure

```
artifacts/
  api-server/   — Express 5 REST API + WebSocket server (port 8080 in dev)
  kds/          — React 19 + Vite frontend (KDS display + management UI)
  mockup-sandbox/ — Isolated Vite server for Canvas/design previews
lib/
  db/           — Drizzle ORM schema + PostgreSQL client (@workspace/db)
  api-spec/     — OpenAPI YAML (single source of truth for the contract)
  api-zod/      — Orval-generated Zod schemas (do not hand-edit)
  api-client-react/ — Orval-generated React Query hooks (do not hand-edit)
scripts/        — Utility scripts (shared, leaf workspace package)
docker/         — Dockerfile.api, Dockerfile.web, nginx config
references/     — Developer reference docs (db.md, server.md, openapi.md)
docs/           — Integration guides (VOLANTE.md, DEVELOPER.md, etc.)
```

## Key rules

### Never do these things
- **Never `cd` before running commands** — always use `--filter` from workspace root.
- **Never run `pnpm dev` at the workspace root** — use `restart_workflow` instead.
- **Never hand-edit files in `lib/api-zod/` or `lib/api-client-react/`** — they are
  generated by Orval from `lib/api-spec/src/openapi.yaml`. Run codegen after spec changes:
  `pnpm --filter @workspace/api-spec run codegen`
- **Never use `console.log` in server code** — use `req.log` in route handlers and the
  `logger` singleton (from `./lib/logger`) in non-request code.
- **Never import from `zod/v4`** in route files or schema files — use `from "zod"`.
- **Never add artifacts to the root `tsconfig.json` references** — that file is for libs only.
- **Never call `broadcast()` when you mean `broadcastToDevice()`** — `broadcast()` sends to
  all connected clients; `broadcastToDevice(id, event)` targets a single screen.
- **Never hard-code `PORT` or `BASE_PATH`** in config files — read from `process.env`.

### Always do these things
- **Read `references/db.md`** before adding DB tables or running migrations.
- **Read `references/server.md`** before adding new API routes.
- **Read `references/openapi.md`** before adding endpoints to the OpenAPI spec.
- **Run `pnpm run typecheck`** after any TypeScript changes and fix all errors.
- **Use `pnpm --filter @workspace/db run push`** to apply schema changes to the local DB.
- **Use `pnpm --filter @workspace/db run push-force`** only when Drizzle asks you to confirm
  destructive schema changes.

## Database workflow

Schema files live in `lib/db/src/schema/`. Each table gets its own file.
After adding or modifying a schema file:
1. Export the new table from `lib/db/src/schema/index.ts`
2. Run `pnpm --filter @workspace/db run push` (or `push-force` if prompted)
3. Import the table in your route via `import { ..., newTable } from "@workspace/db"`

Drizzle-kit config: `lib/db/drizzle.config.ts`
DB client: `lib/db/src/index.ts` (exports `db` + all tables)

## WebSocket event contract

All WS messages follow: `{ type: string, payload: { ... } }`

| Event type | Direction | Description |
|---|---|---|
| `register` | client → server | Client sends its `deviceId` on connect |
| `connected` | server → client | Server acknowledges connection |
| `order_created` | server → client | New order arrived |
| `order_updated` | server → client | Order status changed |
| `order_bumped` | server → client | Order bumped off display |
| `item_status_updated` | server → client | Individual item status change |
| `orders_cleared` | server → client | All orders cleared |
| `kds_config_push` | server → device | Config template pushed to specific display |
| `kds_ping` | server → device | Ping to confirm display is alive |

`broadcastToDevice(deviceId, event)` sends to one device.
`broadcast(event)` sends to all connected clients.

## TypeScript patterns

Orval-generated query hooks use TanStack Query v5. If you see a TypeScript error on
`queryKey` or `enabled`, use the `as any` escape hatch:
```ts
useListDevices({ storeId }, { query: { enabled: !!storeId } as any })
```

Route handlers must be typed as `async (req, res): Promise<void>` and always `return`
after sending a response to prevent "headers already sent" errors.

| `BASE_PATH` | Yes (dev) | Frontend base path (set by workflow config) |

| `ADMIN_PASSWORD` | No | Enables HTTP Basic auth on all `/api` routes |
| `VOLANTE_STATION_MAP` | No | JSON map of Volante terminal IDs → station slugs |

## CI / Docker

CI runs in `.github/workflows/ci.yml`:
1. TypeCheck (libs + leaf packages)
2. Build `@workspace/api-server`
3. Build `@workspace/kds` (Vite)
4. Docker build for both images

Docker images:
- `docker/Dockerfile.api` → API server (Node 24 Alpine, port 3000)
- `docker/Dockerfile.web` → KDS frontend (Nginx, port 80)

Both Dockerfiles copy `package.json` stubs for all workspace packages to avoid
`ERR_PNPM_LOCKFILE_CONFIG_MISMATCH`. If you add a new workspace package, add its
`package.json` stub to both Dockerfiles.

## Common mistakes to avoid

- Adding `pingActive` or other `useState` hooks after conditional logic — hooks must always
  be called in the same order.
- Using `from "zod/v4"` instead of `from "zod"` in route files (causes TS errors).
- Forgetting to export new schema files from `lib/db/src/schema/index.ts`.
- Calling `broadcast()` instead of `broadcastToDevice()` in targeted-push routes.
- Hardcoding `PORT` in `vite.config.ts` — it breaks Docker build stages where PORT is unset.
- Forgetting to `return` after `res.status(...).json(...)` in async route handlers.
